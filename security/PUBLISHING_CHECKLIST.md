# Publishing Checklist

Use this file before adding any new project to the portfolio.

Confirm that all API keys, bearer tokens, client secrets, webhook URLs, account identifiers, database IDs, credential blocks, private prompts, pinned data, lead lists, email addresses, phone numbers, private documents, and generated reports have been removed.

Confirm that the project description explains the business problem without exposing the client or any private operating detail. The case study should focus on the workflow logic, tools used at a high level, technical decisions, your contribution, and the value created.

Confirm that any workflow shown publicly is either a recreated demo or a sanitized explanation. Do not upload raw production workflows from previous employers or clients.

Confirm that all examples use fictitious data when needed. Do not include real names, private companies, internal URLs, social account identifiers, Google Sheet links, Airtable base IDs, Dropbox paths, Slack channels, Telegram bot tokens, WhatsApp phone IDs, CRM records, or media links from production systems.

Confirm that the final README and case studies are written in a way that is clear for recruiters and safe for public review.
